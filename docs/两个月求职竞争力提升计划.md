# 两个月求职竞争力提升计划

## 一、求职定位与目标

### 1.1 岗位定位

主投“机器人学习/具身智能算法工程师”，副投“机器人算法落地/系统工程师”。暂不主投纯控制、纯嵌入式或基础大模型研究岗位。

选择这一路线的依据：

- 当前已经具备 ACT、MuJoCo、LeRobot、数据采集、部署评估和 Replay 微调的完整实验闭环。
- 当前具身智能岗位普遍要求模仿学习、VLA、仿真、数据闭环，并将 ROS2、C++、真机或 Sim2Real 作为重要工程能力证明。
- 两个月内无法同时补齐控制、真机和大规模基础模型训练，采用“一套作品、两版简历”的投入产出比更高。

岗位要求参考：

- [具身强化学习算法工程师](https://career.nankai.edu.cn/correcruit/content/id/113536.html)
- [上海人工智能实验室具身智能操作算法工程师](https://www.shlab.org.cn/joinus/detail/7587270254460455206?jobFunction=&jobType=&keyword=&location=CT_125&mode=social&subject=)
- [具身 VLA 算法工程师](https://www.lumosbot.tech/job-detail-algorithm-vla.html)

### 1.2 两个月后的验收目标

- 一个能够公开复现的机器人学习 GitHub 项目。
- ACT、Replay-ACT、Diffusion Policy 三组同口径实验。
- 一个 ROS2 Humble 仿真部署闭环，其中包含实际 C++ 节点。
- 一段 90 秒演示视频和一份 5 页项目说明。
- 两版一页简历：算法版和系统版。
- 至少完成 60 个定向投递和 6 次模拟面试。

## 二、八周执行安排

### 第 1 周：把现有项目整理成可信作品

- 固化当前 150-seed 实验结果，撰写中英双语案例报告。
- 如实呈现实验结论：Replay 成功率从 86.7% 提升至 88.7%，整体差异不显著；共同成功场景的执行步数降低约 6%；边界场景改善，但局部区域存在退化。
- 加入系统架构图、训练配置、seed 配对方法、初始化复现修复和典型失败案例。
- 提供一键执行命令，覆盖数据检查、训练、单 seed 部署、批量评估和结果汇总。
- 增加以下自动化检查：
  - 相同 seed 的场景复现测试。
  - Replay 采样比例测试。
  - `seed_results.csv` 与 metrics JSON 的一致性测试。
- 大模型文件和数据集使用 Git LFS 或 GitHub Release 管理。
- 清理机器相关绝对路径，确保其他机器可以按 README 运行。

### 第 2～3 周：增加 Diffusion Policy 对照实验

直接使用当前 LeRobot 0.4.4 已提供的 `DiffusionPolicy`，不在本阶段实现 π0、OpenVLA 或 RDT。

- 保持与 ACT 相同的双相机输入、机器人状态、动作空间和 action chunk 配置。
- 使用原始 30 条数据和 5 条补充数据，训练采样比例固定为 80%/20%。
- 先运行 3 个 seed 的冒烟测试，再运行 50 个严格配对 seed。
- GPU 时间允许时，将评估扩展到 150 个配对 seed。
- 与 ACT v5、Replay-ACT 使用完全相同的初始场景、最大步数和成功判断口径。
- 统一输出以下指标：
  - 成功率及置信区间。
  - 同 seed 成败迁移。
  - 共同成功场景的执行步数。
  - 动作平滑度和预测一致性。
  - 失败 seed 与空间位置分布。
- 无论 Diffusion Policy 是否超过 ACT，都公开完整结果，重点展示公平实验、失败分析和工程实现能力。

### 第 4～5 周：完成最小但真实的 ROS2/C++ 闭环

开发环境固定为 Ubuntu 22.04 和 ROS2 Humble。

建立 `vla_policy_bridge` ROS2 包，形成以下数据流：

1. MuJoCo 发布前视图、腕部图像和机器人关节状态。
2. Python 策略节点加载 checkpoint 并执行推理。
3. 策略节点发布原始动作。
4. C++ 安全节点检查动作合法性并发布安全动作。
5. MuJoCo 接收安全动作并完成闭环控制。

采用以下标准话题接口：

- `/camera/front/image_raw`
- `/camera/wrist/image_raw`
- `/joint_states`
- `/policy/raw_action`
- `/policy/safe_action`
- `/policy/reset`

C++ `safety_gate` 节点至少实现：

- 关节位置限位。
- 相邻动作速度变化限制。
- 消息时间戳超时检查。
- NaN 和无穷值检查。
- 非法动作拒绝与错误日志。

同时提供：

- ROS2 launch 文件。
- YAML 配置文件。
- rosbag 录制与回放命令。
- 节点异常退出后的恢复方式。
- 推理延迟及端到端控制延迟的 p50/p95 指标。
- Dockerfile 和一条命令启动的演示脚本。

系统岗位材料中明确写成“ROS2 + MuJoCo 仿真部署闭环”，不声称具备真机部署经验。

### 第 6 周：制作求职材料

#### 算法版简历项目描述

> 基于 LeRobot 与 MuJoCo 构建机械臂 ACT 模仿学习闭环，完成数据采集、训练、Temporal Ensemble 部署、失败诊断及 80/20 Replay 微调；在 150 个严格配对场景上进行统计评估，识别边界场景收益与局部退化，并将共同成功任务的执行步数降低约 6%。

算法版简历突出：

- ACT、Diffusion Policy 和模仿学习原理。
- 数据闭环、Replay 微调与灾难性遗忘分析。
- 配对实验、置信区间和失败模式分析。
- 模型泛化能力与边界场景表现。

#### 系统版简历内容顺序

1. ROS2 消息链路和 C++ 安全节点。
2. 推理服务、配置管理和可复现部署。
3. 仿真环境重置、异常场景拒绝、指标与日志。
4. ACT 和 Diffusion Policy 模型训练。

#### 作品集产物

- 90 秒演示视频：任务演示、典型失败、修复效果和 ROS2 节点图。
- 5 页 PDF：问题、系统、方法、实验和局限。
- README 首屏展示演示视频、核心结果表和一键运行命令。
- 单独增加“我发现并修复的问题”章节，说明：
  - 环境初始化非确定性。
  - 只用 5 条新数据微调导致灾难性遗忘。
  - 50 个 seed 结论被 150 个 seed 结果修正。
  - Replay 微调带来的场景收益与局部退化。

### 第 7～8 周：面试准备与投递

从第 2 周开始投递，不等待项目全部完成。

- 每周定向投递 10～15 个岗位。
- 投递比例：70% 机器人学习/操作算法岗位，30% 算法落地/机器人系统岗位。
- 每完成 20 次投递统计一次回复率。
- 回复率低于 10% 时，优先修改简历首屏、项目标题和岗位关键词。
- 每周至少进行一次算法模拟面试和一次项目深挖模拟面试。

每天安排：

- 45 分钟：Python/C++ 数据结构与算法。
- 45 分钟：坐标变换、正逆运动学、Jacobian、PID 和控制频率。
- 60 分钟：ACT、行为克隆、协变量偏移、Diffusion Policy、Replay 和灾难性遗忘。
- 30 分钟：ROS2 QoS、消息同步、线程、延迟和故障恢复。
- 30 分钟：分别使用 3 分钟和 15 分钟版本讲解当前项目。

## 三、作品验收标准

招聘者应当能够在 5 分钟内确认：

- 项目代码可以运行，而不是只有论文复现截图。
- 候选人能够解释模型为什么成功或失败，而不只是调整参数。
- 实验使用严格配对、置信区间和反例分析，没有夸大 2 个百分点的提升。
- 策略能够接入 ROS2 系统，并通过 C++ 节点实现安全边界。
- 对项目局限有清晰认识，包括没有真机验证和没有大规模多任务泛化。

## 四、两个月内明确不做的事项

- 不从零实现大型 VLA 模型。
- 不同时开展 PPO、SAC、世界模型等多个新方向。
- 不继续进行缺少新问题定义的 TE 参数扫描。
- 不为获得漂亮结果删除失败 seed 或只展示前 50 个 seed。
- 不把纯仿真结果包装为 Sim2Real 或真机部署经验。
- 不追求大量低相关技术关键词，所有新增内容必须服务于目标岗位。

## 五、默认条件

- 当前处于硕士应届求职阶段，计划两个月后集中投递。
- 可以稳定使用 NVIDIA GPU。
- 当前项目可以完整公开到 GitHub。
- 暂时没有真实机械臂，ROS2/C++ 从基础开始补充。
- 当前 ACT 项目继续作为主作品，不另起无关项目。
- 算法岗位是第一优先级，系统岗位只投与策略部署、仿真和机器人学习紧密相关的职位。
